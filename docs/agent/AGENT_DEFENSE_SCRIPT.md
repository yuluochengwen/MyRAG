# Agent 毕设答辩讲解脚本

## 一、3分钟演示版

### 1. 开场（20秒）

- 本系统在现有 RAG 平台上实现了一个可配置 Agent 工作台。
- 目标是让系统不仅能“回答”，还能“展示执行过程”和“形成可评估数据”。

### 2. 页面功能演示（80秒）

- 打开 Agent 页面：`/static/agent-demo.html`。
- 左侧可配置：模型、最大迭代、温度、是否展示步骤。
- 中间对话区：输入问题后，系统显示思考中状态。
- 右侧会话区：支持新建会话、切换会话、清空会话。
- 提问后展示：
  - 用户输入
  - Agent回复
  - 执行步骤（Thought/Action/Observation）
  - 统计信息（迭代次数、步骤数、耗时）

### 3. 技术亮点（50秒）

- 后端接口扩展支持 `llm_model`、`temperature`、`show_steps`。
- `session_id` 真正接入上下文，支持最近多轮记忆。
- 结构化日志可追踪 `iterations/tools_called/duration_ms/success`。
- 前端支持异常恢复，失败后可继续会话，不会白屏。

### 4. 实验结果（30秒）

- 已提供 A/B 实验脚本：
  - A 模式：简洁回复
  - B 模式：Agent工具链+步骤可视化
- 自动输出 JSONL、CSV、Markdown 报告，支撑论文实验章节。
- 本次基线实测（2026-03-19，30题，单次重复）：
  - 两种模式成功率均为 100%。
  - A 模式平均耗时约 10738 ms，B 模式平均耗时约 12336 ms。
  - A 模式平均步骤 0，B 模式平均步骤约 2.2，体现了可解释性增益与时延成本的权衡。

## 二、8分钟答辩版

### 1. 研究问题（1分钟）

- 传统问答页存在两个问题：
  - 过程不可解释，用户只看到最终答案。
  - 难以评估，缺少统一指标与复现实验流程。
- 因此本工作重点是“可解释 + 可评估 + 可落地”。

### 2. 系统设计（2分钟）

- 架构：前端工作台 -> Agent API -> Agent Service(ReAct循环) -> 工具层。
- 前端模块：
  - 运行配置模块
  - 会话管理模块
  - 对话渲染模块
  - 步骤与统计模块
- 后端模块：
  - 请求参数校验与兼容
  - 上下文管理
  - 工具调用执行与异常兜底
  - 指标日志

### 3. 实现要点（2分钟）

- 前端：
  - localStorage 存储会话与活动会话ID
  - `show_steps` 开关控制步骤渲染
  - pending状态控制按钮禁用和状态提示
- 后端：
  - prompt 中注入最近 N 轮历史
  - 在成功、失败、超迭代分支都写入历史
  - 统一返回字段，确保前端稳定消费

### 4. 实验设计（2分钟）

- 数据集：30题，3类（事实问答、计算推理、知识检索）。
- 方案：A/B 双模式，每题重复多次。
- 指标：
  - 成功率
  - 平均耗时与P95耗时
  - 平均迭代与步骤数
  - 主观质量分（脚本启发式占位，可替换为人工评分）
- 结果文件：
  - `agent_eval_ab_metrics.jsonl`
  - `agent_eval_ab_summary.csv`
  - `agent_eval_ab_report.md`

### 5. 本次实测结论（可直接口播，1分钟）

- 时间：2026-03-19。
- 配置：30题数据集、A/B 双模式、每题 1 次重复、同一后端环境。
- 结果：
  - 成功率：A=100%，B=100%。
  - 按类别耗时（ms）：
    - fact：A=6458.4，B=7872.7
    - math：A=5304.0，B=7110.0
    - kb：A=20452.7，B=22023.8
  - 解释性指标：A 平均步骤 0；B 平均步骤为 2.6（fact）、2.0（math）、2.0（kb）。
- 结论：
  - 若强调低时延，A 模式更合适。
  - 若强调可解释过程展示，B 模式更适合用于教学/答辩演示。

### 6. 局限与改进（1分钟）

- 局限：
  - 目前步骤为请求完成后返回，非逐步流式。
  - 会话上下文为进程内存，重启后清空。
  - 质量分当前是启发式，后续可引入人工标注或LLM评审。
- 改进：
  - 增加 SSE 流式步骤推送。
  - 会话落库并支持历史检索。
  - 引入更严格的离线评测集与标注规范。

## 三、常见答辩问答

1. 为什么选择 ReAct？

- 因为 ReAct 兼顾“思考+行动”，天然适合工具调用，且步骤可解释，便于前端可视化。

1. 为什么要做 A/B 对比？

- 只展示单一结果无法证明改进有效；A/B 可以定量说明收益与代价（如延迟）。

1. 如何保证实验公平？

- 使用同一数据集、相同重复次数、相同后端环境，仅改变实验模式参数。

1. 遇到模型不稳定怎么办？

- 通过结构化日志保留失败样本，报告成功率与P95而非只报均值，并提供降级路径。
